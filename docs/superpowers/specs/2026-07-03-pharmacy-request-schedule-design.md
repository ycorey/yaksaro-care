# 약사 요청 관리 — 마감 기준 오늘 중심 아젠다 설계

- **날짜**: 2026-07-03
- **상태**: 승인됨 (스펙 리뷰 대기)
- **대상**: `pharmacy_requests`(마이그레이션 043) · `pharmacy-request-inbox` · `pharmacy/(app)/page.tsx` · `api/pharmacy/request/*` · `lib`(신규 순수 헬퍼)

## 배경 / 문제

약사 대시보드의 "환자 요청함"은 접수순 평면 카드 리스트다. 단골 환자가 늘수록
"언제 어떤 요청이 왔는지 / 약을 **언제까지** 준비해야 하는지"를 체계적으로 볼 수 없다.
현재 `pharmacy_requests`에는 **마감(준비 기한) 필드가 없고**, 시간축 정리·지연 표시도 없다.

## 목표

요청함을 **마감 기준 "오늘 중심 아젠다"**(지연/오늘/내일/이번 주/이후)로 개편해, 약사가
"오늘 뭘 준비하지"를 한눈에 보고, 각 요청의 마감을 조정하며 운영할 수 있게 한다.

## 결정 사항 (브레인스토밍)

- **마감 출처**: 유형별 기본값(전 유형 = 접수일/당일) 자동 설정 + **약사가 조정**.
- **뷰**: 오늘 중심 아젠다(마감 버킷). 월 캘린더 그리드 아님.
- **배치**: 기존 요청함을 이 아젠다로 **교체/업그레이드**(별도 페이지 없음).
- **마감 단위**: **날짜(day)**. 픽업 "오후 3시" 같은 구체 시각은 지금처럼 약사 답변 텍스트가 담당.
- **환자 화면**: 무변경.

## 설계

### 1) 데이터 모델 — 마이그레이션 043

- `pharmacy_requests`에 **`due_date date`** 추가(nullable).
- 기존 행 백필: `due_date = (created_at at time zone 'Asia/Seoul')::date` (마이그레이션 내에서 1회). 이후 신규 행은 아래 API가 항상 세팅 → 실질적으로 non-null 유지.
- **역할분기 트리거(038 `pharmacy_requests_pin_immutable`) 갱신**: 신규 컬럼 `due_date`는 트리거가 손대지 않으면 통과되어 **환자도 변경 가능**한 구멍이 생긴다. 환자 브랜치에 `new.due_date := old.due_date;`를 추가해 **환자는 due_date 변경 불가**, 약사 브랜치는 그대로 통과시켜 **약사만 수정** 가능하게 한다. (공통 pin 컬럼·나머지 로직은 038 그대로 유지)

### 2) 마감 기본값 (요청 생성 시)

- 요청 생성 API(`api/pharmacy/request` — 환자가 요청 보낼 때)에서 INSERT에 `due_date = todayKST()` 세팅. 전 유형 기본 = 당일. (유형별 상이 규칙은 지금 불필요 — 약사 조정으로 흡수. YAGNI)

### 3) 마감 변경 (약사 전용)

- 신규 엔드포인트 **`PATCH /api/pharmacy/request/due`** — body `{ id, due_date }`. user(약사) 토큰 + RLS(내 약국 요청만) + 트리거(약사 브랜치 허용)로 삼중 방어. `due_date`는 `YYYY-MM-DD` 형식 검증.
- 약사 카드에서 **오늘 / 내일 / 직접 날짜 선택**으로 변경.

### 4) 아젠다 뷰 (요청함 개편)

- **순수 헬퍼 `bucketByDue(due, today): Bucket`** 신규 (KST 날짜 문자열 기준, I/O 없음 → 단위테스트):
  - `diff = (due - today)` 일수
  - `diff < 0` → `overdue`(지연) · `diff === 0` → `today`(오늘) · `diff === 1` → `tomorrow`(내일)
  - `2 ≤ diff ≤ 6` → `thisWeek`(이번 주) · `diff ≥ 7` → `later`(이후)
- 서버 컴포넌트(`pharmacy/(app)/page.tsx`)가 활성 요청(status `open`·`acknowledged`)에 `due_date`를 실어 넘김(기존 select에 컬럼 추가). 정렬 = `due_date` 오름차순.
- `pharmacy-request-inbox`를 버킷 아젠다로 개편:
  - 버킷 순서: ⚠️ 지연 → 오늘 → 내일 → 이번 주 → 이후. 비어있는 버킷은 숨김.
  - 각 카드: **마감 배지**(오늘 / 내일 / D-n / ⚠️지연) + **마감 변경**(오늘·내일·날짜) + 기존 답변/전화/확인/완료 그대로.
  - 완료·취소는 버킷에서 빠지고 기존 "최근 처리 N건"(접이식)에 유지.

### 5) 데이터 흐름

```
환자 요청 생성 → due_date = 접수일(KST)  [API INSERT]
약사 대시보드 → 활성요청(+due_date) → bucketByDue로 버킷 분류·정렬 → 아젠다 렌더
약사 마감 변경 → PATCH /request/due → (RLS+트리거 약사쓰기) due_date 갱신 → 아젠다 재분류
환자: due_date 변경 시도 → 트리거가 OLD로 고정(무효)
```

### 6) 엣지 케이스

- `due_date` null(백필 실패 등 이론상) → 아젠다에서 `오늘` 버킷으로 안전 취급(누락 방지).
- 잘못된 날짜 문자열 PATCH → 400.
- 이미 완료/취소된 요청 → 아젠다 버킷 제외(기존 active 필터 유지).

## 테스트

- **단위(`bucketByDue`)**: 경계일 케이스 — 어제(overdue)·오늘(today)·내일(tomorrow)·모레(thisWeek)·+6일(thisWeek)·+7일(later). Node 내장 러너(`node --experimental-strip-types --test`), 무의존성.
- **마이그레이션/트리거 시나리오**(admin+세션): 약사 토큰으로 `due_date` UPDATE 성공, 환자 토큰으로 `due_date` UPDATE가 OLD로 고정(무효)·다른 컬럼(status 취소)만 반영됨.
- `tsc`/`lint`/`build`.
