# MVP 정의 방법론

## MoSCoW 분류 기준

**Must Have (없으면 제품이 존재하지 않는 것):**
사용자가 앱을 설치할 이유가 되는 핵심 가치. 약사로 케어에서는:
- 약을 등록하는 방법 (OCR 또는 검색)
- 상호작용을 확인하는 기능
- 데이터를 안전하게 보관하는 인증

**Should Have (있으면 채택률 크게 향상, 없어도 MVP 가능):**
- OTC 등록 (처방약이 핵심이므로)
- 알레르기 프로필
- 약사 모드 (B2B 성장 엔진이지만 V1 초기엔 선택)

**Could Have (V2로 미뤄도 MVP 가치 유지):**
- 건기식 등록·상호작용
- 복약 캘린더·알림
- 가족 관리

**Won't Have (V3 이후):**
- 카카오 알림톡
- 복약 이행률 AI 분석
- 웨어러블 연동

## 공수 추정 계수 (PharmMatch 기준)

1인 Next.js+Supabase 개발자 생산성 기준:
- 기본 CRUD (인증 포함): 3~5일
- 외부 API 연동 (단순): 1주
- 외부 API 연동 (복잡, 인증 필요): 2~4주
- AI 기능 (GPT Vision OCR): 2~3주 (프롬프트 반복 개선 포함)
- 모바일 최적화 UI: 기능당 +20~30% 추가

## 외부 의존성 선행 조치

외부 API 신청은 개발 Day 1에 시작해야 한다:

| API | 예상 승인 기간 | 대안 |
|-----|--------------|------|
| 공공데이터포털 (식약처) | 즉시~1주 | 없음, 필수 선행 |
| 심평원 DUR API | 4~8주 (협약 필요) | M1: 더미 데이터로 UI 개발, M2: 실 API 연동 |
| GPT-4o API | 즉시 (유료) | 비용 예산 확보 필요 |

## 마일스톤 설계 원칙

각 마일스톤 말에 베타 사용자(약사 1~2명)에게 데모가 가능해야 한다.
"완성"이 아닌 "피드백 가능한 상태"가 마일스톤 기준이다.

## 출시 성공 기준 (정량)

- 베타: 약사 10명 이상 파트너 확보
- Day 30: 약국 소개를 통한 환자 100명 등록
- Day 90: 약사 50명 파트너 + 환자 500명 + 처방전 OCR 정확도 85% 이상

## 기술 위험 Top 3

1. **DUR API 승인 지연**: 심평원 협약이 8주 이상 걸리면 M2 지연. 완화: 공개 약품 데이터로 기본 상호작용 구현, DUR 연동은 M3로 이동.
2. **OCR 정확도**: 한국어 처방전 약품명 인식률. 완화: GPT-4o + 사용자 직접 교정 UI 필수 구현.
3. **규제 해석**: SaMD 해당 시 식약처 인허가 필요. 완화: 표현 가이드 준수 + 법무 검토 선행.
