# Design QA Review — 핸드오프 구현 검증

생성물이 스펙과 실제로 일치하는지 **파일:라인 근거**로 검증한다. 핵심은 "있다/없다"가 아니라 **"스펙 기대값 vs 코드 실제값"의 교차 비교**다.

## 0. 선행 확인

- `design_handoff_yaksaro_care/README.md` — Design Tokens(기대 hex), Regulatory Constraints, 화면 수용 기준
- `_workspace/ui/01_design-system_output.md`, `02_screens_output.md` — 무엇을 구현했다고 주장하는지
- 실제 구현 파일 — 주장과 코드 대조

## 1. 토큰 정합성

- **구 블루 잔존 탐지**: 구현 코드에서 옛 블루가 남아있는지 grep.
  ```
  Grep "#2563eb|#1e40af|#dbeafe|#eff6ff|blue-600|blue-700" (src/app, src/components)
  ```
  남아있으면 그린 토큰으로 전환됐어야 할 자리인지 판단(처방약 아이콘은 blue500 `#4A8FCC`로 의도된 경우 있음 — README 확인).
- **하드코딩 hex 우회**: 화면이 `#0E6E54`를 직접 쓰지 않고 토큰/유틸을 참조하는가.
- **핵심 hex 일치**: green600 `#0E6E54`, green50 `#E3F2EB`, pageBg `#EFEBE2`, neutral900 `#13261F` 등이 globals.css에 정확히 정의됐는가.
- **shadcn 변수 보존**: oklch 기반 `--primary` 등이 삭제되지 않았는가(삭제 시 기존 UI 붕괴).

## 2. 규제 준수 (최우선·차단)

렌더 텍스트에서 금지 표현 탐지:
```
Grep "복용\s*중단|복용을?\s*중단|처방\s*변경|끊으세요|끊으십|위험합니다|진단|투약\s*중지" (src)
```
- 발견 시 **즉시 차단(blocker) 보고** — 다른 항목보다 우선.
- 상호작용 안내가 안전 표현("알려진 상호작용 정보가 있습니다", "약사와 상담")인지 확인.
- 마케팅 카피가 기능 과시(AI 위험분석)가 아니라 문제 해결 중심인지.

## 3. 접근성·실버 UX

- **터치 타겟 ≥44px**: 버튼/토글/탭의 높이·패딩 확인(YCButton size sm36은 보조용만 허용, 주요 액션 44+).
- **폰트 최소 기준**: 본문 18px 지향, 핵심 정보 24px+. 11–13px는 캡션/배지에 한정됐는가.
- **대비**: 텍스트/배경 대비(neutral400 캡션이 충분한가).
- **의사 제시 뷰**: Share 화면 doctor-view가 더 크고 단순한 읽기 전용인가.

## 4. 통합 무결성 (경계면 버그 — 가장 흔한 결함)

- **import 해석**: 참조한 컴포넌트/토큰이 실제 존재하는가(존재하지 않는 `@/components/yc/...` 참조 금지).
- **실데이터 연결**: 프로토타입 더미가 코드에 남지 않았는가.
  ```
  Grep "아모잘탄|리피토|타이레놀.*더미|데모" (src)  // 실데이터인지 하드코딩 더미인지 판별
  ```
  화면이 여전히 기존 Supabase 쿼리/Server Action/API에서 데이터를 받는가(서버 컴포넌트 쿼리 → 렌더 props 경로 추적).
- **타입 무결성**: `npx tsc --noEmit` 통과. 실패 시 출처가 이번 작업인지 기존 코드인지 구분.
- **shape 일치**: 데이터 쿼리가 주는 필드와 컴포넌트가 기대하는 props가 일치하는가(예: 쿼리에 없는 필드를 렌더가 참조 → 런타임 undefined).

## 5. 보고 형식 (`_workspace/ui/03_qa_report.md`)

각 항목을 표로:
| 영역 | 항목 | 상태 | 근거(파일:라인) | 기대 | 실제 | 심각도 |
|---|---|---|---|---|---|---|

- 심각도: **차단**(규제 위반·tsc 실패·깨진 import) / **경고**(토큰 불일치·터치타겟 미달) / **개선**(미세 픽셀차).
- FAIL은 담당 에이전트에 SendMessage로 즉시 회신(수정문 포함), 수정 후 재검증.

## 작업 리듬

- **점진적**: 모듈(토큰/컴포넌트/화면) 완료 통지를 받을 때마다 그 부분만 검증. 전체 끝나고 1회로 미루지 않는다.
- **regression 체크**: 재호출 시 이전 FAIL이 다시 통과하는지부터.

## 흔한 함정

- "컴포넌트 파일이 존재함"만 보고 PASS → 실제 화면이 그 컴포넌트를 쓰는지, props가 맞는지까지 확인.
- tsc 실패를 전부 이번 작업 탓으로 → 기존 코드 유래는 분리 보고.
- 규제 위반을 "경고"로 → 항상 차단.
