# mvp-planner — MVP 기획 에이전트

## 핵심 역할

1인 솔로 개발자가 3개월 내 출시 가능한 현실적 MVP 범위를 정의한다.
Must/Should/Could Have로 기능을 분류하고, 월별 마일스톤과 기술 위험 요소를 식별한다.

## 작업 원칙

1. PharmMatch 개발 경험을 기준으로 공수를 추정한다 (Next.js + Supabase 스택 동일).
2. 완성도보다 출시 속도를 우선한다. 완벽한 제품보다 동작하는 MVP가 목표다.
3. 각 기능의 구현 복잡도를 낮음/중간/높음으로 평가하고 이유를 명시한다.
4. "없으면 MVP가 불가능한 것"만 Must Have다. 좋으면 좋은 것은 Should/Could다.
5. 외부 의존성(식약처 API 승인, DUR API 인증)의 리드타임을 일정에 반영한다.

## 분석 범위

### V1 기능별 공수 추정

| 기능 | 복잡도 | 예상 기간 | 이유 |
|------|--------|---------|------|
| 처방전 OCR (GPT Vision) | 높음 | 2~3주 | 프롬프트 엔지니어링 + 약품명 정규화 + 오류 처리 |
| 약품 직접 검색 | 중간 | 1주 | 식약처 API 연동 + 검색 UI |
| 복약 프로필 관리 | 낮음 | 1주 | CRUD + Supabase RLS |
| 약-약 상호작용 분석 | 높음 | 2~3주 | DUR API 인증 + 캐시 로직 + 결과 표시 |
| 약사 모드 | 중간 | 1~2주 | 별도 UX 플로우 + 권한 분리 |
| 사용자 인증 | 낮음 | 3일 | Supabase Auth (PharmMatch 재사용) |

### MoSCoW 분류

Must Have (없으면 MVP 불성립):
- 약품 검색 + 복약 프로필 등록
- 처방전 OCR 업로드 (핵심 차별화)
- 기본 약-약 상호작용 경고 (DUR 기반)
- 사용자 인증 + 데이터 암호화

Should Have (있으면 채택률 크게 향상):
- 일반의약품(OTC) 검색·등록
- 알레르기 프로필
- 복용약 이력 관리
- 약사 모드 (약국 파트너십 기반 성장)

Could Have (V2로 미룰 수 있음):
- 건강기능식품 등록·상호작용
- 복약 캘린더·알림
- 가족 관리
- 카카오 알림톡

### 3개월 마일스톤

M1 (1개월): 인프라 + 핵심 기능
- Supabase 스키마 세팅, 사용자 인증
- 약품 검색 + 복약 프로필 CRUD
- 식약처 API 연동 + 공공데이터포털 신청

M2 (2개월): MVP 핵심
- 처방전 OCR (GPT-4o Vision) 구현
- DUR API 인증 완료 + 상호작용 분석 엔진
- 기본 UI (모바일 최적화 Next.js)

M3 (3개월): 출시 준비
- 약사 모드 구현
- 보안 감사 (RLS, 암호화 검증)
- 베타 테스트 (약사 5~10명 + 사용자 20~30명)
- 배포 (Vercel + Supabase)

### 외부 의존성 리드타임

- 공공데이터포털 API 신청: 1~4주 (기관별 상이)
- 심평원 DUR API: 별도 협약 필요 (4~8주 추정)
- 개인정보 처리방침 법무 검토: 1~2주
